# 学习笔记：配置与并发

记录练手过程中沉淀下来的设计思路，**与具体项目解耦**，便于以后在其他系统里复用或对比。

---

## 1. 配置：快照与管理分离

### 1.1 核心思想

把配置拆成两层：

| 层 | 职责 | 特点 |
|----|------|------|
| **快照（Snapshot）** | 某一时刻的完整配置视图 | 不可变；创建后只读 |
| **管理器（Manager）** | 加载、持有、替换快照；对外提供读接口 | 可变的是「指针指向哪份快照」 |

Reload 不是原地改 map，而是：**读入新数据 → 构建新快照 → 原子替换当前快照**。

### 1.2 优点

- **读路径简单**：读者要么看到旧快照全文，要么看到新快照全文，没有「改到一半」的 map
- **Reload 失败可回退**：新快照构建失败时，旧快照不受影响
- **读文件可放在锁外**：先在临时结构中加载，成功后再 swap，缩短持锁时间
- **语义清晰**：快照 = 某一版本配置；管理器 = 当前生效版本 + 加载逻辑

### 1.3 缺点与注意点

- **内存**：每次 Reload 一份新拷贝；配置很大、更新很频时需评估
- **成员缓存**：业务若把配置值拷进成员变量，swap 快照**不会**自动更新那些拷贝 → 需 `Rebuild` 或每次从接口读
- **一次请求多 key**：若 Reload 发生在请求中间，可能读到不同版本的 key → 需要 **Snapshot 绑定到请求级**，或 **drain 后再 Reload**
- **类型安全**：KV 全是字符串，上层需 `GetInt` 等封装，避免拼写错误静默失败

### 1.4 接口注入 vs 裸单例

- **推荐**：业务依赖 `ConfigReader` 抽象，构造时注入；生产用 Manager，测试用 Mock
- **裸单例**（业务里到处 `Instance().Get()`）：写起来快，但依赖隐藏、单测难、全局状态污染

单例可以存在，但宜作为 **Manager 的存放方式**，不宜作为 **业务代码的调用方式**。

### 1.5 进程内并发读

常见组合：

- **读写锁 + swap**：`Get` 用读锁；`Reload` 用写锁换指针
- **atomic + shared_ptr**（更轻读路径）：读侧 `atomic_load`，写侧 `atomic_store` 新快照

多 **进程** 之间各有一份 Manager，**不共享**锁；集群一致靠配置中心 + 各节点依次 Reload。

## 2. Gate / Guard：排空后再更新

### 2.1 核心思想

两层配合：

| 层 | 机制 | 解决什么 |
|----|------|----------|
| **Gate（门）** | 服务是否接新请求；在途请求计数 | 节点级「停业整顿」 |
| **Guard（门卫）** | RAII：构造时登记进入，析构时登记离开 | 单次请求生命周期内计数准确 |

更新配置（或其它需要短暂停服的操作）典型顺序：

```text
StopAccepting → 等 in-flight 归零（drain）→ 执行更新 → StartAccepting
```

每个请求入口：`Guard g(gate); if (!g.ok()) 拒绝服务; ...`

### 2.2 优点

- **不会漏减计数**：Guard 析构自动 `Leave`，比手写 `TryEnter`/`Leave` 安全，异常路径也覆盖
- **API 收窄**：`TryEnter`/`Leave` 设为 private，只有 Guard 能调，避免业务乱用
- **与锁同模式**：和 `lock_guard<mutex>` 同一套 RAII 心智，易学
- **更新窗口可控**：配合短超时（如 2s drain），适合「请求短、可短暂拒绝新单」的服务（如推荐链路）

### 2.3 缺点与注意点

- **Guard 禁止拷贝**：一份 Guard = 一个名额，拷贝会导致重复 Leave
- **drain 超时**：超时后仍可能有在途请求 → 需定义强切还是继续等
- **只解决单进程**：Gate 是进程内的；集群靠 LB 摘节点 + 各进程各自 drain
- **不替代快照锁**：drain 减少更新窗口内的业务请求；Reload 实现仍要 swap/锁，二者叠加

### 2.4 嵌套 Guard 类的原因

Guard 作为 Gate 的内部类，表达 **「通行证从属于门」**；对外类型名如 `ServiceGate::Guard`，避免全局命名冲突，也暗示不能脱离 Gate 单独使用。

## 3. 两层「一致性」别混为一谈

| 层级 | 手段 | 保证 |
|------|------|------|
| **进程内** | 快照 swap、读写锁 / atomic | Reload 与 Get 不破坏 map；多线程读安全 |
| **节点 / 实例级** | Gate drain | 更新时尽量没有在途业务 |
| **集群级** | 配置中心、滚动发布、LB 摘流 | 各节点逐步切新版本，而非全集群同秒停服 |

「写者优先的读写锁」防的是 **写者饥饿**（写者排队后新读者不进），不是「写者跳过所有已等读者」。  
CPU L1/L2 由硬件一致性 + 锁/atomic 的内存序保证；要担心的是 **代码里的成员缓存** 和 **跨进程版本差**，不是手工刷 CPU 缓存。

## 4. 参数传递与 C++ 习惯（简记）

| 话题 | 要点 |
|------|------|
| `const T&` vs `T&` vs 按值 | 只读用 `const&`；要改用 `&`；小对象偶尔按值 |
| 拷贝构造 | 必须 `const T&`，按值会无限递归 |
| `explicit` | 单参构造禁止隐式转换 |
| 匿名 namespace | 文件内私有符号，防链接期重名，不是防 `#include` |
| `std::move` | 转移堆上资源所有权，不是「地址搬家到栈」 |

## 5. 演进路线（抽象）

```text
conf 文件 + 注入接口
    → 快照 swap + 读写锁
    → Gate drain + Reload
    → 配置中心 watch + 滚动更新
    → 业务接 DB，conf 只留运维参数
```

练手阶段先把 **快照分离 + 接口注入 + Gate/Guard** 在单进程跑通，再想象多实例与配置中心，顺序自然。
