# 学习笔记：配置与并发

记录练手过程中沉淀下来的设计思路，**与具体项目解耦**，便于以后在其他系统里复用或对比。

---

## 1. 配置：快照与管理分离

### 1.1 核心思想

把配置拆成两层：

| 层 | 职责 | 特点 |
|----|------|------|
| **快照（Snapshot）** | 某一时刻的完整配置视图 | 不可变；创建后只读 |
| **管理器（Manager）** | 加载、持有、替换快照；对外提供读接口 | 可变的是「指针指向哪份快照」 |

Reload 不是原地改 map，而是：**读入新数据 → 构建新快照 → 原子替换当前快照**。

### 1.2 优点

- **读路径简单**：读者要么看到旧快照全文，要么看到新快照全文，没有「改到一半」的 map
- **Reload 失败可回退**：新快照构建失败时，旧快照不受影响
- **读文件可放在锁外**：先在临时结构中加载，成功后再 swap，缩短持锁时间
- **语义清晰**：快照 = 某一版本配置；管理器 = 当前生效版本 + 加载逻辑

### 1.3 缺点与注意点

- **内存**：每次 Reload 一份新拷贝；配置很大、更新很频时需评估
- **成员缓存**：业务若把配置值拷进成员变量，swap 快照**不会**自动更新那些拷贝 → 需 `Rebuild` 或每次从接口读
- **一次请求多 key**：若 Reload 发生在请求中间，可能读到不同版本的 key → 需要 **Snapshot 绑定到请求级**，或 **drain 后再 Reload**
- **类型安全**：KV 全是字符串，上层需 `GetInt` 等封装，避免拼写错误静默失败

### 1.4 接口注入 vs 裸单例

- **推荐**：业务依赖 `ConfigReader` 抽象，构造时注入；生产用 Manager，测试用 Mock
- **裸单例**（业务里到处 `Instance().Get()`）：写起来快，但依赖隐藏、单测难、全局状态污染

单例可以存在，但宜作为 **Manager 的存放方式**，不宜作为 **业务代码的调用方式**。

### 1.5 进程内并发读

常见组合：

- **读写锁 + swap**：`Get` 用读锁；`Reload` 用写锁换指针
- **atomic + shared_ptr**（更轻读路径）：读侧 `atomic_load`，写侧 `atomic_store` 新快照

多 **进程** 之间各有一份 Manager，**不共享**锁；集群一致靠配置中心 + 各节点依次 Reload。

## 2. Gate / Guard：排空后再更新

### 2.1 核心思想

两层配合：

| 层 | 机制 | 解决什么 |
|----|------|----------|
| **Gate（门）** | 服务是否接新请求；在途请求计数 | 节点级「停业整顿」 |
| **Guard（门卫）** | RAII：构造时登记进入，析构时登记离开 | 单次请求生命周期内计数准确 |

更新配置（或其它需要短暂停服的操作）典型顺序：

```text
StopAccepting → 等 in-flight 归零（drain）→ 执行更新 → StartAccepting
```

每个请求入口：`Guard g(gate); if (!g.ok()) 拒绝服务; ...`

### 2.2 优点

- **不会漏减计数**：Guard 析构自动 `Leave`，比手写 `TryEnter`/`Leave` 安全，异常路径也覆盖
- **API 收窄**：`TryEnter`/`Leave` 设为 private，只有 Guard 能调，避免业务乱用
- **与锁同模式**：和 `lock_guard<mutex>` 同一套 RAII 心智，易学
- **更新窗口可控**：配合短超时（如 2s drain），适合「请求短、可短暂拒绝新单」的服务（如推荐链路）

### 2.3 缺点与注意点

- **Guard 禁止拷贝**：一份 Guard = 一个名额，拷贝会导致重复 Leave
- **drain 超时**：超时后仍可能有在途请求 → 需定义强切还是继续等
- **只解决单进程**：Gate 是进程内的；集群靠 LB 摘节点 + 各进程各自 drain
- **不替代快照锁**：drain 减少更新窗口内的业务请求；Reload 实现仍要 swap/锁，二者叠加

### 2.4 嵌套 Guard 类的原因

Guard 作为 Gate 的内部类，表达 **「通行证从属于门」**；对外类型名如 `ServiceGate::Guard`，避免全局命名冲突，也暗示不能脱离 Gate 单独使用。

## 3. 两层「一致性」别混为一谈

| 层级 | 手段 | 保证 |
|------|------|------|
| **进程内** | 快照 swap、读写锁 / atomic | Reload 与 Get 不破坏 map；多线程读安全 |
| **节点 / 实例级** | Gate drain | 更新时尽量没有在途业务 |
| **集群级** | 配置中心、滚动发布、LB 摘流 | 各节点逐步切新版本，而非全集群同秒停服 |

「写者优先的读写锁」防的是 **写者饥饿**（写者排队后新读者不进），不是「写者跳过所有已等读者」。  
CPU L1/L2 由硬件一致性 + 锁/atomic 的内存序保证；要担心的是 **代码里的成员缓存** 和 **跨进程版本差**，不是手工刷 CPU 缓存。

## 4. 参数传递与 C++ 习惯（简记）

| 话题 | 要点 |
|------|------|
| `const T&` vs `T&` vs 按值 | 只读用 `const&`；要改用 `&`；小对象偶尔按值 |
| 拷贝构造 | 必须 `const T&`，按值会无限递归 |
| `explicit` | 单参构造禁止隐式转换 |
| 匿名 namespace | 文件内私有符号，防链接期重名，不是防 `#include` |
| `std::move` | 转移堆上资源所有权，不是「地址搬家到栈」 |

## 5. 演进路线（抽象）

```text
conf 文件 + 注入接口
    → 快照 swap + 读写锁
    → Gate drain + Reload
    → 配置中心 watch + 滚动更新
    → 业务接 DB，conf 只留运维参数
```

练手阶段先把 **快照分离 + 接口注入 + Gate/Guard** 在单进程跑通，再想象多实例与配置中心，顺序自然。

## 6. GatedConfigReloader：配置 + Gate 的唯一入口（本仓库演进）

原先 Gate、ConfigReader 分散在 main / 各 server 里调；现在收成 **一个对外类**：

| 能力 | 入口 |
|------|------|
| 启动加载 conf | `GatedConfigReloader::Init(path)` |
| 读 key | `RequireString` / `RequireSizeT`（**无默认值**，缺 key 或非法即失败） |
| 请求准入 | `GatedConfigReloader::RequestGuard`（内部走 ServiceGate，业务不 include gate） |
| 热更新 | `Reload(validator)`：StopAccepting → drain → swap 快照 → 校验 → StartAccepting |

**Reload 不属于 main**：挂在 `RankingGrpcServer::Reload()`，main 里 SIGHUP 只触发 `server.Reload()`。

**Bazel 封装**：`//src/common:common` 的 `hdrs` 只暴露 `gated_config_reloader.h` + `types.h`；`config_manager` / `service_gate` 等在 `config_internal` 里且 **visibility 禁止外部 deps**，从构建上避免业务绕过入口。

与 §2 的关系：Gate/Guard 思想不变，只是 **对外 API 收窄成 GatedConfigReloader**，避免 server 各自摸 ConfigReader 和 ServiceGate。

## 7. gRPC：Channel / Stub / Context（今日主线）

进程内联调改成 **Ranking 独立进程 `ranking_main`** + client demo 远程调 RPC，顺带把传输层概念捋清。

### 7.1 三层对象

| 对象 | 是什么 | 生命周期 |
|------|--------|----------|
| **Channel** | 到 `host:port` 的**逻辑连接管理器**（重连、多路复用），≠ 固定一条 TCP 四元组 | 长期复用 |
| **Stub** | 某种 **proto Service** 的调用入口（Ranking / Video 各一种） | 长期复用，多线程可共用 |
| **ClientContext** | **单次 RPC** 的超时、metadata、取消 | 每次 `Rank()` 新建 |

```text
RankingGrpcClient
  channel_  →  127.0.0.1:50051
  stub_     →  RankingService::Stub
  Rank():   ClientContext ctx; stub_->Ranking(&ctx, ...);
```

### 7.2 RPC 和 HTTP 的关系

- **RPC**：编程模型（像调本地函数）
- **gRPC**：RPC 框架（proto + stub/server）
- **HTTP/2**：gRPC 默认**传输层**（path 形如 `/package.Service/Method`）
- **TCP**：HTTP/2 底下；**一个 server port** 上可挂多个 RPC，靠 path 区分，不是每个 RPC 一个 port

### 7.3 并发怎么来的

- 多线程共用一个 **Stub** + 各自 **ClientContext** ✅
- 同一条 TCP 上 HTTP/2 **多 stream** 并发多个 RPC ✅
- 再建多个同类型 Stub **不会**增加并发能力；多个 Stub 通常是因为**多种 Service 接口**

### 7.4 端口：client 临时口 vs server 监听口

- Server：`ranking_address=127.0.0.1:50051` 写在 conf 里，进程 **bind 50051**
- Client：`connect` 时 OS 分配**临时源端口**（如 54321），不是 gRPC「占坑」；连接关了就释放
- 业务只需关心 conf 里的 **server 地址**，不用管 client 临时 port

### 7.5 服务端：`Impl`、线程、耗时

| 点 | 结论 |
|----|------|
| **RankingServiceImpl** | `Impl` = Implementation；继承 proto 生成的 `RankingService::Service`，填 RPC 逻辑 |
| **`server_->Wait()`** | 阻塞当前线程；独立进程里 main 线程 Wait 即可 |
| **demo 里曾用 std::thread 跑 server** | 同进程又要当 client 才需要；拆成 `ranking_main` 后不需要 |
| **`sleep_for` 在 Rank 里** | 阻塞的是 **gRPC worker 线程**，不是 main；模拟排序耗时 |
| **日志加锁** | 多 RPC 并发 + 直接 `cout/cerr` 时，用 mutex 保证一行不乱；上日志库后通常可去掉 |

## 8. 今日其它 C++ 触点（简记）

| 话题 | 要点 |
|------|------|
| `std::move(address)` | 按值接参后 move 进成员，少一次 string 拷贝 |
| `mutable std::mutex` | `Rank() const` 里要打日志加锁，mutex 状态不算业务语义变化 |
| `RequireString` / 无 default | conf 缺 key 就报错退出，不用 `Get(key, 默认值)` 静默兜底 |
| 独立二进制 | `ranking_main` 只跑 server；`browse_demo` 10 个 Agent 线程并发调 RPC，看总耗时是否 ~1～2s 而非 10s+ |
